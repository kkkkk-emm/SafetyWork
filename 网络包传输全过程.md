# 网络包传输全过程模拟

---

## 一、系统运行主线

Client 先找 AS 证明用户身份，AS 校验账号密码后签发 `TGT` 和 `KcTgs`；Client 再拿着 `TGT` 找 TGS 申请访问目标 GS 的 `Service Ticket` 和 `KcGs`；最后 Client 拿着 `Service Ticket` 接入 GS，GS 验证成功后生成 `sessionId`。从 `GS_AUTH_OK` 返回 `sessionId` 开始，房间、准备、开始对战、输入同步、快照校正、断线重连都由 GS 处理。

数据库方面，MySQL/AuthDB 只保存两类数据：

```text
1. user_account
   保存用户账号、密码摘要、PBKDF2 参数、账号状态 status、登录代数 login_gen。

2. security_event_log
   记录低频安全事件，例如 REGISTER、LOGIN_SUCCESS、LOGIN_FAIL、CHANGE_PASSWORD、
   TICKET_EXPIRED、REPLAY_BLOCKED 等。
```

不进入数据库的数据包括：

```text
TGT、Service Ticket、KcTgs、KcGs
sessionId、roomId、房间成员、准备状态
INPUT、SNAPSHOT、玩家坐标、投射物、空投、战斗事件
AS/TGS/GS 地址和端口
K_TGS、K_GS、RSA 公私钥
```

# 【阶段一】用户注册流程

这里模拟玩家 A 注册账号 `linhai`。

## 第 1 步：ClientA -> AS

发送的 JSON 包：

```json
{
  "type": "REGISTER_REQ",
  "clientId": "cli-a-001",
  "payload": "Base64(RSA({\"username\":\"linhai\",\"password\":\"123456Aa!\"}))"
}
```

这个包的作用：

ClientA 向 AS 提交注册信息。用户名和密码不直接明文放在顶层，而是先用 RSA 加密，再 Base64 编码，放进 `payload`。

接收方收到后做什么：

AS 先检查 `type = REGISTER_REQ` 是否属于自己支持的认证消息，再检查必需字段 `clientId` 和 `payload`。然后 AS 解码并使用 RSA 私钥解密 `payload`，得到：

```json
{
  "username": "linhai",
  "password": "123456Aa!"
}
```

AS 执行内部数据库操作：

```text
AS -> user_account：
- 按 username = "linhai" 查询是否已经存在。
- 若不存在，生成 password_salt、pbkdf2_iter、password_hash。
- 插入新用户记录：
  username = "linhai"
  password_hash = PBKDF2(password, password_salt, pbkdf2_iter)
  password_salt = ...
  pbkdf2_iter = 100000
  login_gen = 0
  status = 1
  created_at = 当前时间
  updated_at = 当前时间

AS -> security_event_log：
- 写入 REGISTER 事件。
- result = 1 表示成功，result = 0 表示失败。
- 记录 client_id = "cli-a-001"。
```

发送后产生的状态变化：

```text
ClientA：
- 暂时只知道自己发起了注册请求。
- 还没有 TGT。
- 还没有 KcTgs。
- 还不能访问 TGS / GS。

AS：
- 解析并校验注册材料。
- 写入 user_account。
- 写入 security_event_log。

MySQL/AuthDB：
- user_account 新增一条用户记录。
- security_event_log 新增一条 REGISTER 审计记录。

TGS：
- 无变化。

GS：
- 无变化。
```

## 第 2 步：AS -> ClientA

发送的 JSON 包：

```json
{
  "type": "REGISTER_REP",
  "payload": "{\"ok\":true,\"userId\":10001}"
}
```

这个包的作用：

AS 告诉 ClientA：注册成功，并返回新建账号的 `userId`。

接收方收到后做什么：

ClientA 解析顶层 JSON，发现 `type = REGISTER_REP`，再对 `payload` 字符串执行一次 JSON 解析，得到：

```json
{
  "ok": true,
  "userId": 10001
}
```

发送后产生的状态变化：

```text
ClientA：
- 知道账号注册成功。
- 可以记录 userId = 10001 用于界面展示。
- 仍然没有 TGT。
- 注册成功不等于登录成功。

AS：
- 完成本次注册响应。

MySQL/AuthDB：
- user_account 已保存账号认证数据。
- security_event_log 已记录注册事件。

TGS：
- 无变化。

GS：
- 无变化。
```

# 【阶段二】用户登录并获取 TGT 流程

现在 ClientA 使用账号 `linhai` 登录，目标是从 AS 获取 `TGT` 和 `KcTgs`。

## 第 1 步：ClientA -> AS

发送的 JSON 包：

```json
{
  "type": "AS_REQ",
  "clientId": "cli-a-001",
  "payload": "Base64(RSA({\"username\":\"linhai\",\"password\":\"123456Aa!\",\"nonce\":\"n1\"}))"
}
```

这个包的作用：

ClientA 向 AS 证明“我是账号 `linhai` 的合法用户”，并请求 AS 签发 `TGT`。其中 `nonce = n1` 用来防止旧响应被重放。

接收方收到后做什么：

AS 检查 `type / clientId / payload`，解密 `payload`，得到：

```json
{
  "username": "linhai",
  "password": "123456Aa!",
  "nonce": "n1"
}
```

AS 执行内部数据库操作：

```text
AS -> user_account：
- 按 username = "linhai" 查询用户。
- 读取 user_id、password_hash、password_salt、pbkdf2_iter、login_gen、status。
- 检查 status 是否为 1。
- 使用 password_salt 和 pbkdf2_iter 重新计算 PBKDF2 摘要。
- 对比摘要是否等于 password_hash。

若登录成功：
- 更新 last_login_at。
- 可按实现策略决定是否递增 login_gen。
  - 如果设计为“每次登录使旧票据失效”，则 login_gen += 1。
  - 如果设计为“改密/强制下线才使旧票据失效”，则登录时不递增 login_gen。
- 生成 KcTgs。
- 使用服务端本地配置中的 K_TGS 加密生成 TGT。

AS -> security_event_log：
- 写入 LOGIN_SUCCESS 或 LOGIN_FAIL 审计事件。
- 记录 username、user_id、client_id、remote_addr、reason。
```

TGT 明文结构为：

```json
{
  "userId": 10001,
  "clientId": "cli-a-001",
  "kcTgs": "...",
  "loginGen": 17,
  "exp": 1776679210100
}
```

注意：ClientA 不解析 TGT 明文，ClientA 只缓存并转发 `Base64(TGT)`。

发送后产生的状态变化：

```text
ClientA：
- 等待 AS_REP。
- 本地仍没有 TGT / KcTgs。

AS：
- 完成账号密码验证。
- 生成 KcTgs。
- 生成 TGT。
- 准备返回受保护的用户部分。

MySQL/AuthDB：
- user_account 被读取。
- user_account.last_login_at 可能被更新。
- user_account.login_gen 可能按策略更新。
- security_event_log 写入 LOGIN_SUCCESS 或 LOGIN_FAIL。

TGS：
- 无变化。

GS：
- 无变化。
```

## 第 2 步：AS -> ClientA

发送的 JSON 包：

```json
{
  "type": "AS_REP",
  "ticket": "Base64(TGT)",
  "payload": "{\"salt\":\"Base64(16B)\",\"iter\":100000,\"part\":\"Base64(DES(Kuser,{\\\"nonce\\\":\\\"n1\\\",\\\"kcTgs\\\":\\\"...\\\",\\\"exp\\\":1776679210100,\\\"loginGen\\\":17}))\"}"
}
```

这个包的作用：

AS 返回两部分内容：

```text
1. ticket = Base64(TGT)
   用于后续提交给 TGS。

2. payload.part = Base64(DES(Kuser, {...}))
   用用户密码派生出的 Kuser 保护，里面有 nonce、KcTgs、exp、loginGen。
```

接收方收到后做什么：

ClientA 先解析顶层 JSON。然后从 `payload` 字符串中取出 `salt / iter / part`。ClientA 使用自己的密码、`salt` 和 `iter` 派生 `Kuser`，再用 `Kuser` 解密 `part`，得到：

```json
{
  "nonce": "n1",
  "kcTgs": "...",
  "exp": 1776679210100,
  "loginGen": 17
}
```

ClientA 校验：

```text
part.nonce 是否等于自己刚才发出的 n1。
part 是否能被 Kuser 正确解密。
TGT 是否存在。
kcTgs 是否存在。
```

发送后产生的状态变化：

```text
ClientA：
- 缓存 clientId = cli-a-001。
- 缓存 TGT = Base64(TGT)。
- 缓存 KcTgs。
- 缓存 TGT 过期时间 exp。
- 可以进入 TGS_REQ 阶段。

AS：
- 登录流程完成。

MySQL/AuthDB：
- user_account 保留当前 login_gen、status、last_login_at。
- security_event_log 已记录登录事件。

TGS：
- 尚未收到任何请求。

GS：
- 尚未收到任何请求。
```

# 【阶段三】申请 Service Ticket 并接入 GS 流程

这一阶段分两段：

```text
ClientA -> TGS：用 TGT 换 Service Ticket。
ClientA -> GS：用 Service Ticket 换 sessionId。
```

## 第 1 步：ClientA -> TGS

发送的 JSON 包：

```json
{
  "type": "TGS_REQ",
  "clientId": "cli-a-001",
  "ticket": "Base64(TGT)",
  "auth": "Base64(DES(KcTgs,{\"ts\":1776650500000,\"nonce\":\"n2\"}))",
  "payload": "Base64(DES(KcTgs,{\"service\":\"game/ws@10.0.0.12:9100\",\"nonce\":\"n3\"}))"
}
```

这个包的作用：

ClientA 告诉 TGS：“我已经通过 AS 登录，手里有 AS 签发的 `TGT`，现在我要访问 `game/ws@10.0.0.12:9100` 这个 GS 服务。”

接收方收到后做什么：

TGS 检查顶层字段：

```text
type 必须是 TGS_REQ。
clientId 必须存在。
ticket 必须存在。
auth 必须存在。
payload 必须存在。
```

然后 TGS 做以下验证：

```text
1. 使用服务端本地配置中的 K_TGS 解开 ticket，得到 TGT 明文。
2. 检查 TGT.clientId 是否等于顶层 clientId。
3. 检查 TGT.exp 是否过期。
4. 从 user_account 查询当前 user_id 对应的 login_gen 和 status。
5. 检查 status 是否为 1。
6. 检查 TGT.loginGen 是否等于 user_account.login_gen。
7. 用 TGT 里的 KcTgs 解密 auth。
8. 检查 auth.ts 是否处于允许时间窗口。
9. 检查 auth.nonce 在窗口内是否未重复使用。
10. 用 KcTgs 解密 payload。
11. 检查 payload.service 是否等于服务端本地配置中允许访问的 GS 标识。
```

TGS 内部数据库操作：

```text
TGS -> user_account：
- 根据 TGT.userId 查询 login_gen 和 status。
- GS 服务地址、端口和 K_GS 均来自 TGS 服务端本地配置。

TGS -> security_event_log：
- 如果 TGT 过期，写入 TICKET_EXPIRED。
- 如果检测到 nonce/ts 重放，写入 REPLAY_BLOCKED。
- 如果 login_gen 不一致，写入 TICKET_INVALIDATED 或类似安全事件。
- 成功签发 Service Ticket 时，可写入低频授权成功事件，例如 TGS_ISSUE_SUCCESS。
```

发送后产生的状态变化：

```text
ClientA：
- 等待 TGS_REP。
- 仍持有 TGT / KcTgs。

TGS：
- 验证 TGT。
- 验证 Authenticator。
- 使用本地固定配置确认目标 GS 服务。
- 使用本地固定配置中的 K_GS 生成 Service Ticket。
- 生成 KcGs。
- 记录短期 nonce/ts 防重放缓存。

MySQL/AuthDB：
- user_account 被读取 login_gen 和 status。
- security_event_log 可写入低频安全事件。

AS：
- 无变化。

GS：
- 无变化。
```

## 第 2 步：TGS -> ClientA

发送的 JSON 包：

```json
{
  "type": "TGS_REP",
  "ticket": "Base64(ServiceTicket)",
  "payload": "Base64(DES(KcTgs,{\"nonce\":\"n3\",\"kcGs\":\"Base64(8-byte-key)\",\"exp\":1776657700100}))"
}
```

这个包的作用：

TGS 把访问 GS 所需的 `Service Ticket` 返回给 ClientA，同时用 `KcTgs` 加密返回 `KcGs`。

`Service Ticket`明文如下：

```json
{
  "userId": 10001,
  "clientId": "cli-a-001",
  "service": "game/ws@127.0.0.1:9100",
  "kcGs": "Base64(8-byte-key)",
  "loginGen": 17,
  "exp": 1776657700100
}
```

接收方收到后做什么：

ClientA 使用本地缓存的 `KcTgs` 解密 `payload`，得到：

```json
{
  "nonce": "n3",
  "kcGs": "Base64(8-byte-key)",
  "exp": 1776657700100
}
```

ClientA 校验：

```text
payload.nonce 是否等于 TGS_REQ 中发送的 n3。
是否成功拿到 KcGs。
是否成功拿到 Service Ticket。
```

发送后产生的状态变化：

```text
ClientA：
- 缓存 Service Ticket = Base64(ServiceTicket)。
- 缓存 KcGs。
- 缓存 Service Ticket 过期时间 exp。
- 可以连接 GS 并发送 GS_AUTH。

TGS：
- 完成一次服务票据签发。
- 短期记录 nonce/ts 防重放缓存。
- 可写入 security_event_log 记录低频授权事件。

MySQL/AuthDB：
- security_event_log 可有一条授权成功或失败记录。

AS：
- 无变化。

GS：
- 仍未建立业务 session。
```

## 第 3 步：ClientA -> GS

发送的 JSON 包：

```json
{
  "type": "GS_AUTH",
  "clientId": "cli-a-001",
  "ticket": "Base64(ServiceTicket)",
  "auth": "Base64(DES(KcGs,{\"ts\":1776650510000,\"nonce\":\"n4\"}))"
}
```

这个包的作用：

ClientA 拿着 `Service Ticket` 接入 GS，请求从“已授权访问 GS”升级为“已建立业务会话”。

接收方收到后做什么：

GS 检查：

```text
1. type 是否为 GS_AUTH。
2. clientId 是否存在。
3. ticket 是否存在。
4. auth 是否存在。
```

然后 GS 验证：

```text
1. 使用服务端本地配置中的 K_GS 解开 Service Ticket。
2. 检查 Service Ticket.service 是否绑定当前 GS。
3. 检查 Service Ticket.clientId 是否等于顶层 clientId。
4. 检查 Service Ticket.exp 是否过期。
5. 从 user_account 查询当前 userId 的 login_gen 和 status。
6. 检查 status 是否为 1。
7. 检查 Service Ticket.loginGen 是否等于 user_account.login_gen。
8. 取出 Service Ticket 中的 KcGs。
9. 用 KcGs 解密 auth。
10. 检查 auth.ts 是否在 30 秒窗口内。
11. 检查 auth.nonce 是否重复。
```

GS 内部数据库操作：

```text
GS -> user_account：
- 根据 ServiceTicket.userId 查询 login_gen 和 status。

GS -> security_event_log：
- 如果 Service Ticket 过期，写入 TICKET_EXPIRED。
- 如果检测到重放，写入 REPLAY_BLOCKED。
- 如果 login_gen 不一致，写入 TICKET_INVALIDATED。
- 如果 GS_AUTH 成功，可写入 GS_AUTH_SUCCESS。
```

发送后产生的状态变化：

```text
ClientA：
- 等待 GS_AUTH_OK。
- 暂时还不能发送房间/对战消息。

GS：
- 验证 Service Ticket。
- 验证 Authenticator。
- 建立连接对象。
- 准备生成 sessionId。

MySQL/AuthDB：
- user_account 被读取 login_gen 和 status。
- security_event_log 可写入低频安全事件。
- 不保存 sessionId。
- 不保存房间状态。

AS：
- 无变化。

TGS：
- 无变化。
```

## 第 4 步：GS -> ClientA

发送的 JSON 包：

```json
{
  "type": "GS_AUTH_OK",
  "sessionId": "sess-a-9001",
  "payload": "Base64(DES(KcGs,{\"ts\":1776650510000,\"nonce\":\"n4\",\"exp\":1776657700100}))"
}
```

这个包的作用：

GS 确认 ClientA 认证成功，并返回后续业务阶段使用的 `sessionId`。

接收方收到后做什么：

ClientA 使用 `KcGs` 解密 `payload`，检查：

```text
payload.ts 是否等于自己刚才发出的 ts。
payload.nonce 是否等于自己刚才发出的 n4。
payload.exp 是否仍有效。
sessionId 是否存在。
```

发送后产生的状态变化：

```text
ClientA：
- 缓存 sessionId = sess-a-9001。
- 进入 GS 业务阶段。
- 之后房间、心跳、输入、重连都使用 sessionId。

GS：
- 建立 sessionId -> connection 的映射。
- 建立 sessionId -> clientId/userId 的会话表记录。
- 这些状态都在 GS 内存中，不进入 MySQL。
- 该连接通过认证门禁，允许进入房间流程。

MySQL/AuthDB：
- security_event_log 可记录 GS_AUTH_SUCCESS。
- 不保存 sessionId、roomId、房间成员、对战状态。

AS：
- 无变化。

TGS：
- 无变化。
```

# 【阶段四】创建房间、加入房间、准备、开始对战流程

假设 ClientA 已经拿到：

```text
sessionId = sess-a-9001
userId = 10001
username = linhai
```

ClientB 也已经完成同样的登录、TGS、GS_AUTH 流程，并拿到：

```text
sessionId = sess-b-9002
userId = 10002
username = xiaowang
```

## 第 1 步：ClientA -> GS，创建房间

发送的 JSON 包：

```json
{
  "type": "ROOM_CREATE_REQ",
  "sessionId": "sess-a-9001",
  "auth": "Base64(DES(KcGs,{\"ts\":1776650520000,\"nonce\":\"n_room_create\",\"sessionId\":\"sess-a-9001\"}))"
}
```

这个包的作用：

ClientA 请求 GS 创建一个新房间。`sessionId` 用于告诉 GS 当前请求属于哪个业务会话，`auth` 用 `KcGs` 加密，里面包含当前时间戳 `ts`、一次性随机数 `nonce` 和本次请求绑定的 `sessionId`。这样 GS 不只检查 `sessionId` 是否存在，还会验证发送方是否真的持有该会话对应的 `KcGs`。

接收方收到后做什么：

GS 检查：

```text
GS 检查：
1. type 是否为 ROOM_CREATE_REQ。
2. sessionId 是否存在。
3. sessionId 是否已经通过 GS_AUTH。
4. 使用该 sessionId 对应的 KcGs 解密 auth。
5. 检查 auth.sessionId 是否等于顶层 sessionId。
6. 检查 auth.ts 是否处于允许时间窗口内。
7. 检查 auth.nonce 是否未被重复使用。
8. 检查该 session 是否已经在其他房间中。
9. 当前是否允许创建房间。
```

发送后产生的状态变化：

```text
ClientA：
- 等待 ROOM_CREATE_REP / ROOM_STATE。

GS：
- 生成 roomId = A7K9Q2。
- 创建房间对象。
- 将 ClientA 加入房间。
- 设置 ClientA 为房主。
- 分配 slotNo / localClientId。
- 房间状态为 WAITING。
- 以上全部是 GS 内存状态。

MySQL/AuthDB：
- 无写入。

AS / TGS：
- 无变化。
```

## 第 2 步：GS -> ClientA，返回创建结果

发送的 JSON 包：

```json
{
  "type": "ROOM_CREATE_REP",
  "sessionId": "sess-a-9001",
  "roomId": "A7K9Q2",
  "payload": "Base64(DES(KcGs,{\"ok\":true,\"sessionId\":\"sess-a-9001\",\"roomId\":\"A7K9Q2\",\"ts\":1776650521000,\"nonce\":\"n_room_create_rep\"}))"
}
```

这个包的作用：

GS 告诉 ClientA 房间创建成功，并返回 `roomId`。其中顶层 `roomId` 用于客户端快速识别房间，`payload` 使用 `KcGs` 加密，里面再次绑定 `sessionId`、`roomId`、`ts` 和 `nonce`，用于防止创建结果被伪造或篡改。

接收方收到后做什么：

```text
ClientA 检查：
1. type 是否为 ROOM_CREATE_REP。
2. 顶层 sessionId 是否等于本地 sessionId。
3. 顶层 roomId 是否存在。
4. 使用 KcGs 解密 payload。
5. 检查 payload.ok 是否为 true。
6. 检查 payload.sessionId 是否等于顶层 sessionId。
7. 检查 payload.roomId 是否等于顶层 roomId。
8. 检查 payload.ts 是否处于允许时间窗口内。
```

接收方收到后做什么：


# 【阶段六】断线重连流程

现在模拟 ClientA 在对战中断线。断线前 ClientA 本地缓存：

```text
clientId = cli-a-001
sessionId = sess-a-9001
roomId = A7K9Q2
Service Ticket = Base64(ServiceTicket)
KcGs = Base64(8-byte-key)
lastProcessedSeq = 108
未确认输入队列 = seq > 108 的输入
```

## 第 1 步：GS 检测到 ClientA 断线

这一阶段不是 Client 发 JSON，而是 GS 的连接管理检测到 WebSocket 断开。

GS 做的事情：

```text
1. 标记 sess-a-9001 对应连接断开。
2. 不立即删除 session。
3. 将该 session 放入重连宽限状态。
4. 房间内玩家 online 字段可暂时变为 false。
5. 保留 roomId、matchId、lastProcessedSeq、比赛上下文。
```

发送后产生的状态变化：

```text
ClientA：
- 本地检测到连接断开。
- 保留 Service Ticket / KcGs / old sessionId / lastProcessedSeq。
- 准备发起 RECONNECT_REQ。

GS：
- sess-a-9001 进入重连宽限期。
- 保留房间和对战上下文。
- 可向 ClientB 广播 ROOM_STATE，标记 ClientA offline。
- 重连宽限状态在 GS 内存中。

ClientB：
- 看到对方暂时掉线或等待重连。

MySQL/AuthDB：
- 默认无变化。
- 如果需要审计异常断线，可写 security_event_log。
```

## 第 2 步：GS -> ClientB，广播房间/在线状态变化

发送的 JSON 包：

```json
{
  "type": "ROOM_STATE",
  "sessionId": "sess-b-9002",
  "roomId": "A7K9Q2",

}
```

这个包的作用：

GS 告诉仍在线的 ClientB：ClientA 目前断线，但还没有被立即移除。该状态仍然属于权威房间状态，所以使用 ClientB 的 `KcGs` 加密并绑定 `sessionId / roomId / ts / nonce`。

接收方收到后做什么：

```text
ClientB 检查：
1. type 是否为 ROOM_STATE。
2. 顶层 sessionId 是否等于 sess-b-9002。
3. 顶层 roomId 是否等于 A7K9Q2。

5. 检查 payload.type / payload.sessionId / payload.roomId 是否与顶层一致。
6. 检查 payload.ts 是否处于允许时间窗口内。

```

发送后产生的状态变化：

```text

- 本地房间/战斗 UI 标记 ClientA offline。

GS：
- 继续保留 ClientA 的重连上下文。

ClientA：
- 正在重连。

MySQL/AuthDB：
- 无变化。
```

## 第 3 步：ClientA -> GS，发起 RECONNECT_REQ

发送的 JSON 包：

```json
{
  "type": "RECONNECT_REQ",
  "clientId": "cli-a-001",
  "sessionId": "sess-a-9001",
  "ticket": "Base64(ServiceTicket)",
  "auth": "Base64(DES(KcGs,{\"type\":\"RECONNECT_REQ\",\"clientId\":\"cli-a-001\",\"sessionId\":\"sess-a-9001\",\"roomId\":\"A7K9Q2\",\"ts\":1776650535000,\"nonce\":\"n5\"}))",
  "payload": "Base64(DES(KcGs,{\"type\":\"RECONNECT_REQ\",\"clientId\":\"cli-a-001\",\"sessionId\":\"sess-a-9001\",\"roomId\":\"A7K9Q2\",\"lastProcessedSeq\":108,\"ts\":1776650535000,\"nonce\":\"n5\"}))"
}
```

这个包的作用：

ClientA 告诉 GS：“我是之前的 `sess-a-9001`，我还持有原来的 `Service Ticket` 和 `KcGs`，我最近确认到的输入是 `lastProcessedSeq = 108`，请恢复我的业务会话。”`auth` 和 `payload` 都用 `KcGs` 加密，并共同绑定 `type / clientId / sessionId / roomId / ts / nonce`，防止重连请求被拼接、篡改或重放。

接收方收到后做什么：

GS 检查顶层字段：

```text
type = RECONNECT_REQ。
clientId 存在。
sessionId 存在。
ticket 存在。
auth 存在。
payload 存在。
```

GS 验证逻辑：

```text
1. 使用服务端本地配置中的 K_GS 解开 Service Ticket。
2. 检查 ticket.clientId 是否等于 cli-a-001。
3. 检查 ticket.service 是否绑定当前 GS。
4. 检查 ticket.exp 是否仍有效。
5. 从 user_account 查询当前 login_gen 和 status。
6. 检查 status 是否为 1。
7. 检查 ticket.loginGen 是否等于 user_account.login_gen。
8. 用 KcGs 解密 auth。
9. 检查 auth.type 是否等于 RECONNECT_REQ。
10. 检查 auth.clientId 是否等于顶层 clientId。
11. 检查 auth.sessionId 是否等于顶层 sessionId。
12. 检查 auth.roomId 是否等于 A7K9Q2。
13. 检查 auth.ts 是否在 30 秒窗口内。
14. 检查 auth.nonce = n5 是否未重放。
15. 用 KcGs 解密 payload。
16. 检查 payload.type / payload.clientId / payload.sessionId / payload.roomId 是否与顶层和 auth 一致。
17. 检查 payload.nonce 是否等于 auth.nonce。
18. 读取 payload.lastProcessedSeq = 108。
19. 检查 sess-a-9001 是否仍处于重连宽限期。
20. 恢复 sess-a-9001 对应的连接、房间、比赛上下文。
```

GS 内部数据库操作：

```text
GS -> user_account：
- 根据 ServiceTicket.userId 查询 login_gen 和 status。

GS -> security_event_log：
- 若 Service Ticket 过期，写入 TICKET_EXPIRED。
- 若 nonce/ts 重复，写入 REPLAY_BLOCKED。
- 若 login_gen 不一致，写入 TICKET_INVALIDATED。
- 若重连成功，可写入 RECONNECT_SUCCESS。
- 若重连超时，可写入 RECONNECT_TIMEOUT。
```

发送后产生的状态变化：

```text
ClientA：
- 等待 RECONNECT_REP。

GS：
- 验证旧票据和新 Authenticator。
- 恢复旧 sessionId。
- 将新 WebSocket 连接绑定到 sess-a-9001。
- 恢复 roomId = A7K9Q2。
- 恢复 phase = PLAYING。
- 恢复 lastProcessedSeq 上下文。
- 这些恢复状态都在 GS 内存中。

MySQL/AuthDB：
- user_account 被读取 login_gen 和 status。
- security_event_log 可写入重连相关低频安全事件。
- 不保存 sessionId、roomId、lastProcessedSeq、比赛上下文。

ClientB：
- 等待 GS 广播对方 online。
```

## 第 4 步：GS -> ClientA，返回 RECONNECT_REP

发送的 JSON 包：


这个包的作用：

接收方收到后做什么：

ClientA 用 `KcGs` 解密 `payload`，然后检查：

```text

2. 顶层 sessionId 是否等于本地 sessionId。

5. payload.sessionId 是否等于顶层 sessionId。

14. 在收到 SNAPSHOT 后基于权威状态回放未确认输入。
```

发送后产生的状态变化：

```text

- 房间成员 online = true。
- 重连宽限状态结束。

MySQL/AuthDB：
- 可写入 security_event_log 记录 RECONNECT_SUCCESS。
- 不保存对战状态。

ClientB：
- 等待 ROOM_STATE 或 SNAPSHOT 得知对方恢复在线。
```

# 最终调试时间线

```text
1. Client -> AS：REGISTER_REQ
2. AS -> user_account：写入用户账号、密码摘要、盐值、PBKDF2 参数、login_gen、status
3. AS -> security_event_log：写入 REGISTER 审计事件
4. AS -> Client：REGISTER_REP

5. Client -> AS：AS_REQ
6. AS -> user_account：查询账号、认证参数、login_gen、status
7. AS -> security_event_log：写入 LOGIN_SUCCESS / LOGIN_FAIL
8. AS -> Client：AS_REP，返回 TGT + KcTgs 受保护部分

9. Client -> TGS：TGS_REQ，携带 TGT + Authenticator + service
10. TGS -> user_account：读取 login_gen / status，判断 TGT 是否仍有效
11. TGS -> 本地配置：读取固定 GS 标识与 K_GS
12. TGS -> security_event_log：按需写入 TICKET_EXPIRED / REPLAY_BLOCKED / 授权事件
13. TGS -> Client：TGS_REP，返回 Service Ticket + KcGs

14. Client -> GS：GS_AUTH，携带 Service Ticket + Authenticator
15. GS -> user_account：读取 login_gen / status，判断 Service Ticket 是否仍有效
16. GS -> security_event_log：按需写入 GS_AUTH_SUCCESS / TICKET_EXPIRED / REPLAY_BLOCKED
17. GS -> Client：GS_AUTH_OK，返回 sessionId

18. ClientA -> GS：ROOM_CREATE_REQ，使用 KcGs 生成 auth 绑定 sessionId / ts / nonce
19. GS 内存：创建 roomId、房间对象、成员列表、房主状态
20. GS -> ClientA：ROOM_CREATE_REP + ROOM_STATE，使用 KcGs 加密 payload

21. ClientB -> GS：ROOM_JOIN_REQ，使用 KcGs 生成 auth 绑定 sessionId / roomId / ts / nonce
22. GS 内存：加入成员、分配 Client2 / slotNo
23. GS -> ClientB：ROOM_JOIN_REP，使用 KcGs 加密 payload
24. GS -> ClientA / ClientB：ROOM_STATE，分别使用各自 KcGs 加密 payload

25. ClientA / ClientB -> GS：ROOM_READY_REQ，使用 KcGs 加密 payload 绑定 ready / sessionId / roomId / ts / nonce
26. GS 内存：更新 ready 状态
27. GS -> ClientA / ClientB：ROOM_READY_REP + ROOM_STATE，使用 KcGs 加密 payload

28. 房主 ClientA -> GS：ROOM_START_REQ，使用 KcGs 生成 auth 绑定 sessionId / roomId / ts / nonce
29. GS 内存：创建 matchId、初始化玩家权威状态、输入缓冲区、lastProcessedSeq
30. GS -> ClientA / ClientB：ROOM_START_REP，分别使用各自 KcGs 加密 payload

31. ClientA / ClientB -> GS：INPUT，使用 KcGs 加密 payload 绑定输入内容 / seq / tick / ts / nonce
32. GS 内存：缓存输入并推进权威逻辑帧
33. GS -> ClientA / ClientB：SNAPSHOT，分别使用各自 KcGs 加密 payload
34. Client 根据 lastProcessedSeq 删除已确认输入，并回放未确认输入

35. ClientA -> GS：HEARTBEAT_REQ，使用 KcGs 生成 auth 绑定 sessionId / ts / nonce
36. GS -> ClientA：HEARTBEAT_REP，使用 KcGs 加密 payload 返回对应 nonce
37. GS -> ClientA / ClientB：RESULT，分别使用各自 KcGs 加密 payload

38. ClientA 断线
39. GS 内存：保留 sessionId、roomId、match、lastProcessedSeq，进入重连宽限期
40. GS -> ClientB：ROOM_STATE，使用 ClientB 的 KcGs 加密 payload，标记 ClientA offline
41. ClientA -> GS：RECONNECT_REQ，携带 Service Ticket + auth + payload，使用 KcGs 绑定 clientId / sessionId / roomId / lastProcessedSeq / ts / nonce
42. GS -> user_account：读取 login_gen / status，判断 Service Ticket 是否仍有效
```
